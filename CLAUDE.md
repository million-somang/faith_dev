# 작업 지침

## 최우선 규칙 (이 프로젝트 절대 규칙)

- 코드를 작성할 때 시키는 것 이외에 다른 코드는 건들지 않도록 해줘.
- 어떤 일이 있어도 DB에 있는 데이터는 삭제하지 말아줘.

아래 가이드라인(Karpathy Guidelines 기반)은 위 두 규칙을 더 잘 지키기 위한 작업 방식이다.

## 1. 코딩 전에 생각하기 (Think Before Coding)

추측하지 말고, 모르는 건 숨기지 말고, 트레이드오프를 드러낸다.

- 가정을 명시한다. 불확실하면 물어본다.
- 해석이 여러 가지면 임의로 하나 고르지 말고 제시한다.
- 더 간단한 방법이 있으면 말한다. 필요하면 반대 의견도 낸다.
- 불분명하면 멈추고, 무엇이 헷갈리는지 짚고, 물어본다.

## 2. 단순함 우선 (Simplicity First)

문제를 푸는 최소한의 코드만. 미리 만들어 두는(speculative) 코드는 금지.

- 요청한 것 이상의 기능 추가 금지.
- 한 번만 쓰는 코드에 추상화 금지.
- 요청하지 않은 "유연성"·"설정 가능성" 금지.
- 일어날 수 없는 상황에 대한 에러 처리 금지.
- 200줄 짜리가 50줄로 될 수 있으면 다시 쓴다.
- 스스로 물어본다: "시니어 엔지니어가 이걸 과하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경 (Surgical Changes)

꼭 필요한 곳만 건드리고, 내가 만든 흔적만 정리한다. (= 최우선 규칙)

기존 코드를 수정할 때:
- 옆에 있는 코드·주석·포맷을 "개선"하지 않는다.
- 고장 안 난 걸 리팩터링하지 않는다.
- 내 취향과 달라도 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 삭제하지 말고 언급만 한다.

내 변경으로 고아(orphan)가 생기면:
- 내 변경 때문에 안 쓰이게 된 import·변수·함수만 제거한다.
- 시키지 않은 한, 원래 있던 죽은 코드는 제거하지 않는다.

기준: 바뀐 모든 줄이 사용자의 요청과 직접 연결되어야 한다.

## 4. 목표 기반 실행 (Goal-Driven Execution)

성공 기준을 정의하고, 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터링" → "전후로 테스트가 통과하는지 확인한다"

여러 단계 작업이면 간단한 계획을 먼저 밝힌다:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있으면 독립적으로 반복할 수 있다. 모호한 기준("일단 되게 해줘")은 계속 되묻게 만든다.
